# Outreach And Trust Map

This document defines how SAB should seek trust without hype.

## The Rule

Do not ask serious people to endorse SAB.

Ask them to break it.

## First Contact Targets

| Group | Why they matter | First ask |
| --- | --- | --- |
| MCP server maintainers | They understand tool surfaces and tool-risk reality. | "What would make `standing.fetch` useful before an agent calls a tool?" |
| A2A implementers | They understand delegation and agent-to-agent handoff risk. | "Where should a standing lease attach to an agent card or task result?" |
| AGNTCY contributors | They understand agent discovery, identity, messaging, observability, and evaluation. | "How should standing remain directory-adjacent without becoming a rival directory?" |
| OpenTelemetry practitioners | They understand trace semantics and vendor-neutral telemetry. | "Which SAB attributes are useful, and which are noise?" |
| OpenSSF/Sigstore/SLSA practitioners | They understand artifact provenance and supply-chain evidence. | "Where does provenance stop, and where does adversarial standing begin?" |
| OWASP GenAI contributors | They understand agentic risk taxonomies. | "Which challenge types are missing or mislabeled?" |
| NIST/RMF-literate reviewers | They understand risk framing and public-sector language. | "Where does SAB overclaim authority?" |
| AI security researchers | They understand adversarial failure. | "How would you game witness quorum and standing leases?" |
| Investigative technologists | They understand public evidence and decision-useful reporting. | "What evidence path would make a dossier citable?" |
| Affected-community advocates | They understand consent, extraction, and real-world harm. | "Where does this process create burden or false legitimacy?" |

## First Message Shape

Use this structure:

1. One sentence: SAB is trying to make agent claims challengeable.
2. One concrete artifact: standing standard, dossier, or schema.
3. One narrow ask: find one thing that is misleading, gameable, harmful, or unusable.
4. One promise: sustained challenges will be published and linked to revisions.

Do not send:

- grand claims about becoming the future of agents;
- requests for endorsement;
- vague partnership asks;
- token, marketplace, or certification language;
- "AI for good" framing without a specific evidence path.

## Trust-Building Ladder

### Level 0: Public Readability

Artifacts are public, source-grounded, and written in normal language.

### Level 1: Challengeability

Every claim has a challenge path.

### Level 2: Correction

Sustained challenges change the artifact.

### Level 3: External Use

An external project uses a standing lease to make a narrow decision.

### Level 4: External Refusal

An external party cites a SAB refusal, not just a positive standing.

### Level 5: Forkability

Another group forks the method or evidence bundle and reaches either the same or a different conclusion.

## Collaboration Boundaries

SAB should accept:

- adversarial review;
- evidence contribution;
- schema feedback;
- implementation pilots;
- challenge dossiers;
- governance critique.

SAB should not accept:

- paid favorable conclusions;
- private veto over public findings;
- funder control of standing labels;
- secrecy clauses that block evidence citation;
- community participation without consent and benefit;
- "endorsement" requests disguised as pilots.

## First Five Outreach Moves

1. Publish the first package/tool trust dossier draft and ask maintainers what it gets wrong.
2. Ask one A2A implementer where standing metadata belongs.
3. Ask one OpenTelemetry practitioner to review the proposed SAB attributes.
4. Ask one supply-chain security reviewer to attack the provenance requirements.
5. Ask one public-interest reviewer to flag community extraction and legal-risk issues.

## Success Signal

The first success signal is not applause.

The first success signal is a serious critique that forces SAB to narrow a claim.
