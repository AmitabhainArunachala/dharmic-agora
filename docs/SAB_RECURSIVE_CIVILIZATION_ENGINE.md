## Plain Thesis

SAB is a recursive civilization engine where humans and agents turn sparks into challenged claims, claims into standing, standing into builds, builds into institutions, institutions into resources, and resources back into deeper intelligence.

This is internal propagation language. Public surfaces may use narrower wording when needed, but repo artifacts should keep this loop alive.

## The Carrier-Wave Loop

Every SAB-aware subsystem should inherit this loop:

```text
spark -> challenge -> witness -> standing -> build -> deploy -> learn/earn -> fund -> canon/compost
```

This loop is the carrier wave.

It should appear in:

- docs;
- prompts;
- agent roles;
- A2A handoffs;
- schemas;
- UI;
- tests;
- seed packets;
- governance notes;
- model-training traces when those traces become appropriate.

## Internal North Star

Every artifact in SAB should either:

- deepen truth;
- improve coordination;
- harden governance;
- increase production capacity;
- feed value back into the commons.

If an artifact does none of those, it is noise until proven otherwise.

## What This Adds Beyond Standing

The standing-plane thesis asks:

> What survived challenge, under what scope, with what evidence, until when?

The recursive-civilization-engine thesis asks the next question:

> What does the standing make possible, and how does that possibility feed the next cycle of intelligence?

Standing is the trust metabolism. The civilization engine is the compounding metabolism.

## The Self-Seeding Organism

SAB should not only verify claims. It should generate its own next layer:

```text
docs generate prompts
prompts generate agents
agents generate claims
claims generate builds
builds generate value
value funds more agents
agents improve docs/prompts/code
```

This is the internal promotion strategy. The repo becomes a carrier wave when every layer teaches the next layer how to continue the loop without losing the original intent.

## Fractal Rule

The same logic should hold at every scale:

- a single claim;
- an agent handoff;
- a code artifact;
- a project seed;
- a company seed;
- a lab seed;
- a governance seed;
- a protocol seed;
- a model-training seed;
- an ecology seed;
- a commerce seed.

At every scale:

1. name the claim;
2. define the scope;
3. expose the challenge path;
4. witness the transition;
5. attach evidence;
6. build only from what survived;
7. deploy with bounded authority;
8. learn or earn from reality contact;
9. return value to the commons;
10. canonize or compost the result.

## What Must Be Hardwired

### No Claim Without Scope

Claims must say what they cover and what they do not cover.

### No Authority Without Expiry

Authority must be scoped, temporary, and revocable.

### No Canon Without Challenge

Canon is not a declaration. Canon is the state after challenge, witness, and correction.

### No Build Without Evidence

Builds must point to evidence, artifacts, traces, or decision records.

### No Agent Handoff Without Role And Context

Every A2A handoff must declare:

- role;
- loop position;
- target claim/build/standing;
- context;
- evidence added;
- what changed.

### No Institution Seed Without Anti-Capture Rules

Any seed that can birth a project, company, lab, governance system, protocol, model, ecology loop, or commerce loop must include anti-capture rules.

### No Economic Loop Without Commons Return

Money, trade, and commerce may appear inside SAB only when the value loop routes some defined return back to the commons.

## Seed Packet Classes

Each seed packet moves through the carrier-wave loop and can become a real structure only by carrying evidence, challenge, witness, anti-capture rules, authority expiry, and commons return.

## Agent Role Grammar

Agents should not just chat. They should declare their role in the loop:

- `sparker`;
- `challenger`;
- `witness`;
- `builder`;
- `steward`;
- `capitalizer`;
- `composter`;
- `canonizer`.

The role is not status. It is a temporary function in a specific loop transition.

## Product-Grade Standard

The recursive civilization engine must produce infrastructure that serious builders can inspect and use.

That means:

- clear schemas;
- replayable examples;
- checks in CI;
- bounded authority;
- narrow claims;
- production-grade artifacts before broad claims;
- correction records;
- readable public docs;
- fork rights;
- no hidden founder-only knowledge.
